02 / What the registry stores

Two tiers. Privacy lives in the split.

The registry can be public, replicated, or anchored to a ledger — so it can never hold raw private values. It holds VIN-grade public fields and cryptographic commitments. Raw values stay in custody.

Public · lookup-able by serial

VIN-grade tier

  • Passport exists for (product, serial)
  • Product model + public passport id
  • Purchase year
  • Current owner pseudonym key
  • Number of transfers + timeline
  • Assurance level (self / receipt / channel)
PII-free by construction → safe even if fully enumerated, exactly like a VIN.
Committed · raw values in custody

Sensitive tier

  • Raw serial number
  • The owner's real identity (key↔person)

03 / Who an owner is

One reusable key.
Custody climbs, identity stays.

An owner is a persistent identity — one keypair that signs across many passports. The registry sees a pseudonymous key; hifisync privately maps it to a person. The same public key rides up the custody ladder.

TIER 0
Custodial
hifisync holds the key for you (platform KMS, Privy-style). Crypto is fully hidden. recovery = account recovery — no chain event.
TIER 1
Device-bound
The key moves into your device's secure enclave. recovery = guardian-authorized rotation, on-chain & transparent.
TIER 2
Sovereign
You hold the key outright. recovery = k-of-n guardians (social / MPC) — no lock-in to hifisync. Ultra tier.

Payoff: because identity is one key, a single recovery re-binds your whole collection at once — and correlation across your gear is the accepted, pseudonymous cost.

One unit ⇒ one passport, ever.

The subject id is derived deterministically from the unit, so registration and "reuse what's in custody" are the same operation: mint-or-claim on the anchor. No manufacturer cooperation required.

Mint · assurance is layered

A plain serial can be cloned — so a passport records provenance as claimed, never anti-counterfeit on its own. Assurance is the honest dial, and it strengthens as trusted parties join.

05 / Change of hands

Two-party transfer,
with a custody fallback.

Only the current owner can hand a passport on. The buyer must accept — and if they won't, the seller still gets off the record. Prior owners stay in the append-only chain forever (that history is the value).

06 / Proof & disclosure

The VIN-check link is a
verifiable presentation.

"Reveal full details" and "prove I'm the current owner to a buyer" are one primitive — selective disclosure bound to a signature from the current controller. Field-level: show the serial & service history, hide the price.

At mint
commit each field
commit = hash(value ‖ salt)
Owner
build a short-lived link
Chosen openings + signature over {anchor, nonce, expiry}
Buyer
verify
Operator-rendered now; self-verifiable by anyone later.

The verifier checks four things

01Signature is valid
02That key is the current head of the chain — current owner, not a past one
03Each opening hashes to its commitment on the registry
04The link hasn't expired

Because the commitments are embedded from day one, the same signed bundle that hifipass renders today becomes verifiable by any third party — with zero trust in the operator — the moment keys move to users. No migration.

07 / Lost the key

The custodian as a
fenced guardian.

Recovery is the one sanctioned exception to "only the current controller signs" — which makes it a theft vector too. So the design is really about guardrails. Default policy: {hifisync}, 1-of-1.

01 Identity check
The custodian confirms you're the account owner — off-chain — before acting.
02 Time-lock
A recovery is announced & delayed (~7 days). If the old key resurfaces, it can cancel. This stops silent theft.
03 On-chain & transparent
The recovery is itself an append — visible & provable forever. A malicious rotation can't hide.

One recovery re-binds the reusable identity across all the owner's passports at once. Today it's just account recovery — the on-chain guardian primitive sits dormant until self-custody, exactly like custody_change does now.
